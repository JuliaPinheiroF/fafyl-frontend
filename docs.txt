# Contexto do Projeto: FAFYL — Recomendador de Cursos
## Visão geral
FAFYL é um sistema de recomendação de cursos universitários baseado em perfil de interesse do usuário. O usuário responde um quiz com suas habilidades/interesses e características incompatíveis consigo mesmo; o sistema filtra, pontua e ranqueia os cursos disponíveis retornando uma lista ordenada por afinidade. O nome do projeto vem do mascote da aplicação, o Capelinho — um personagem visual que muda de aparência de acordo com o curso ou preferência do usuário, configurado manualmente pelo frontend.
---
## Stack
- Backend: Java com Spring Boot, Spring Security (JWT), Spring Data JPA
- Banco de dados: Relacional (schema abaixo), com campos JSON para listas e coordenadas
- Autenticação: JWT sem refresh token; existe um endpoint dedicado que renova o token usando o próprio access token ainda válido

## Fluxo de autenticação
### Login — POST /auth/login
1. Cliente envia { email, password }
2. Controller busca o User por email no banco
3. Valida senha
4. Chama JwtService.generateToken(user) → JWT assinado
5. Retorna { token }
### Requisição autenticada
- Todo endpoint protegido passa pelo filtro do Spring Security
- O filtro chama JwtService.validateToken(jwt) → extrai claims
- Se válido, injeta o userId no contexto de segurança
### Renovação de token — POST /auth/renew
1. Cliente envia o access token ainda válido no header
2. Backend valida o token atual (deve estar válido — não aceita expirado)
3. Gera um novo JWT com nova expiração
4. Retorna { novoToken }
> Não existe refresh token separado. A renovação usa o próprio access token enquanto ainda não expirou.

## Endpoint principal — POST /fafyl
### Input (corpo da requisição)
```json
{
  "abilities": ["Memorização", "Empatia", "Ergonomia", "Sustentabilidade"],
  "cantBe":    ["Averso a normas", "Desorganizado financeiramente"]
}
```
> As abilities e o cantBe vêm sempre do corpo da requisição, não de um perfil salvo. O userId é extraído do JWT no header.
### Lógica do FafylService (pipeline em stream)
```java
stream<Course>
  // 1. Filtro cantBe: remove cursos cuja ability bata com qualquer item do cantBe do usuário
  .filter(course -> course.getAbilities().stream().noneMatch(quiz.getUserCantBe()::contains))
  // 2. Scoring: conta interseção entre abilities do quiz e abilities do Course
  .map(course -> new Fafyl(course.compare(quiz.getAbilities()), course))
  // 3. Descarta cursos sem nenhum match
  .filter(fafyl -> fafyl.incidence() > 0)
  // 4. Ordena do maior para o menor score
  .sorted(Comparator.comparingInt(Fafyl::incidence).reversed())
  .toList()
```
*Course.compare(List<String> abilities)**
```java
return (int) abilities.stream()
    .filter(this.abilities::contains)
    .count();
```
Retorna o número de abilities do quiz que estão presentes nas abilities do curso.
*Fafyl** é um DTO com { int incidence, Course course }.
### Após o pipeline
1. Salva o histórico automaticamente: { userId, abilities[], cantBe[], list_curso (Course[]), data }
   - As abilities substituem completamente qualquer registro anterior (não há merge)
2. Retorna 200 OK com List<Fafyl> ranqueada
### Participantes do fluxo
```
Cliente → Controller → FafylService → CourseRepo (streamAll)
                    ↘               ↘ HistoricoRepo (save)
```

## Regras de negócio resumidas
| regra | detalhe |
|---|---|
| cantBe filtra antes do ranking | Se qualquer ability do curso estiver na lista cantBe do usuário, o curso é eliminado |
| incidence > 0 filtra após scoring | Cursos sem nenhuma ability em comum com o quiz não aparecem no resultado |
| Ranking por score descendente | Maior número de abilities em comum = melhor posição |
| Histórico é snapshot imutável | Salvo automaticamente; registra exatamente o que foi enviado e o que foi retornado |
| Capelinho é cosmético | Não influencia nenhum cálculo; escolhido manualmente pelo usuário no frontend |
| Sem merge de abilities | Cada chamada ao /fafyl substitui o registro anterior do histórico daquele usuário |
| Habilidades: Palavras chave relacionadas a esse curso (analitico, cálculo, escritório, home-office, ajudar os outros, dinheiro, resolver conflitos, criatividade, pessoas, sistemas) |
| Não pode ser: Voltado para o que a pessoa não gosta (pessoas, times, hierarquização, longa jornada de trabalho, longo tempo de estudo, periculosidade, insalubridade) |
| Cada curso tem que ter a lista de habilidades necessárias para o curso no campo habilidades e detalhes do curso (que podem tirar candidatos) na parte do não pode ser |
